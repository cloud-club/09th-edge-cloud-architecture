# 1. 주요 개념도 및 아키텍처

전체 구조는 Central SOC/Cloud, Enterprise Edge Node, Endpoint Agent의 3계층으로 나누어 설계합니다. Central은 위협지표·온톨로지·대규모 LLM 처리를 담당하고, Edge는 로컬 Telemetry의 선처리·요약·정책 적용을 수행합니다.

## 1.1. 전체 아키텍처

아래 이미지는 Central, Edge, Endpoint가 어떤 역할을 나누어 갖는지 보여주는 전체 구조도입니다. Central은 공통 지식과 고비용 분석을 담당하고, Edge는 현장 가까이에서 빠른 선처리와 정책 적용을 담당합니다.

!edge_security_architecture_diagram.png

위 아키텍처는 모든 보안 판단을 Edge에 몰아넣는 구조가 아니라, 처리 비용과 실시간성에 따라 Central과 Edge의 역할을 분리하는 구조입니다. 가벼운 이벤트 선별과 로컬 정책 적용은 Edge에서 수행하고, 복잡한 LLM 분석이나 장기 상관분석은 Central에서 수행합니다.

## 1.2. 데이터 흐름도

아래 흐름도는 보안 데이터가 어디서 생성되고, 어느 단계에서 요약·비식별화·분석되는지를 나타냅니다. 특히 주요 보안지표는 Central에서 수집한 뒤 하위 Edge로 전달되는 방향을 기본값으로 둡니다.

!edge_security_data_flow_diagram.png

데이터 흐름은 상향 Telemetry와 하향 보안지표 배포로 구분합니다. Endpoint와 Edge에서 발생한 이벤트는 요약·비식별화 후 Central로 전달하고, Central에서 수집한 CVE·MITRE ATT&CK·위협 인텔리전스는 정책 또는 지식 형태로 Edge에 배포합니다.

# 2. 프로젝트 기획 구체화

이 섹션에서는 실제 구현 범위를 Telemetry 수집, Edge 처리 범위, Central SOC 부하관리로 나누어 구체화합니다. 설계 기준은 데이터 민감도, 실시간성, 모델 처리 비용, 중앙 거버넌스 필요성입니다.

## 2.1. 수집 Telemetry 결정

수집 Telemetry는 운영 맥락을 설명하는 일반 행위 데이터와 즉시 대응이 필요한 의심행위 데이터로 나눕니다. 이 구분에 따라 실시간 전송, 배치 전송, Edge 선처리 여부를 결정합니다.

### 2.1.1. 시스템 행위 관련

시스템 행위 데이터는 단독으로는 위협이 아닐 수 있지만, 사용자·장치·프로세스·시간 정보와 결합되면 이상행위 판단의 기준선이 됩니다.

- 시스템 로그 (이벤트 수집기 기반 수집)
- 로그인 로그 (실시간 기록, 배치전송)
- 설치 프로그램 목록 (배치기록 및 전송)
- 연결장치 목록 (실시간 기록, 배치전송)

위 항목들은 보안 이벤트의 기본 맥락을 구성하는 저위험·상시 수집 데이터입니다. 실시간성이 낮은 정보는 배치 전송하고, 로그인·장치 연결처럼 이벤트성이 강한 정보는 Edge에서 우선 선별한 뒤 Central로 전달합니다.

### 2.1.2. 악성 의심행위 관련

악성 의심행위 데이터는 탐지 즉시 우선순위를 높여 처리합니다. Edge에서는 빠른 요약과 위험도 초안을 만들고, Central에서는 더 넓은 지식 기반으로 최종 판단을 보강합니다.

- 부서단위 비인가 소프트웨어 실행로그 (실시간 기록, 실시간 전송)
- 방화벽 개방로그 (실시간 기록, 실시간 전송)

악성 의심행위는 대응 지연이 직접적인 위험으로 이어질 수 있으므로 Edge에서 빠르게 탐지·요약합니다. 단, 차단·격리 같은 고위험 조치는 Central 정책과 관리자 승인 흐름을 거쳐 SOAR와 연계하는 방향으로 설계합니다.

## 2.2. Edge 기능 구현 범위 결정

Edge 구현 범위는 “모든 분석을 현장에서 끝내는 것”이 아니라, 지연시간·개인정보·네트워크 비용을 줄이기 위한 선별 처리 계층으로 정의합니다.

- 온톨로지 DB: 사용자 및 행위 간 상호상관관계 지식정보를 DB로 구성, 기업 내 행위에 대한 설명이 가능하도록 정보기반 마련
- 데이터 대치필터: 사용자 및 기업정보 식별이 불가하도록 대치하는 filter layer 구성
- NAC, SIEM, SOAR, ZTNA 솔루션 연계: 각 보안장치 별 로그 및 ICMP 프로토콜 수집 엔드포인트를 구성, 중앙집중화된 데이터 관리 및 상관관계 분석 기반 마련

Edge 기능은 원본 데이터를 모두 중앙으로 보내지 않기 위한 로컬 처리 계층입니다. 다만 온톨로지의 기준 지식과 주요 보안지표 수집은 Central에서 관리하고, Edge는 필요한 부분을 캐시하거나 정책 형태로 받아 실행하는 구조가 적합합니다.

## 2.3. Central SOC 부하관리 방안 수립

Central은 CVE, NIST, MITRE ATT&CK, Threat Intelligence 같은 주요 보안지표를 수집·정제하고, 이를 온톨로지 기반 지식 저장소로 관리합니다. 하위 Edge 노드에는 STIX/TAXII 또는 별도 API를 통해 필요한 지표와 정책을 전달하는 방식을 우선 검토합니다.

LLM 처리 또한 Central과 Edge로 분산합니다. 대규모 모델이 필요한 취약점 분석, 장기 상관분석, 복잡한 대응 판단은 Central에서 처리하고, Edge는 실시간 경량 판단과 로컬 요약에 집중해 중앙 부하와 네트워크 비용을 줄입니다.

# 3. 핵심기능 PoC

현재 PoC는 OpenSOAR의 기반이 되는 보안정보 수집 자동화와 대시보드 확인에 초점을 둡니다. 이후 단계에서는 CVE Radar Collector를 확장해 Central 지표 수집, Edge 정책 배포, SOAR 연계를 순차적으로 붙이는 방향입니다.
